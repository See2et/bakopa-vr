# Requirements Document

## Introduction

PeerSession のダイヤル処理にリトライ制御を導入し、一時的な輸送層エラーから利用者を保護する。これにより CLI フラグ `--max-retries` と `--retry-backoff-ms` が意図した堅牢性を発揮し、ユーザーは安定したセッション接続を体験できる。

## Alignment with Product Vision

本機能は steering/product.md で掲げる「ピア優先の信頼性」を具体化する施策であり、P2P セッションが局所的なネットワーク障害下でも継続を試みることで成功率を高める。また、リトライ情報を観測可能にすることで、可観測性向上というプロダクト原則にも合致する。

## Requirements

### Requirement 1

**User Story:** CLI オペレーターとして、接続準備のためにダイヤルコマンドを実行した際、設定された回数と待機時間で自動的に再試行してほしい。そうすれば短時間の障害でセッション確立が中断されない。

#### Acceptance Criteria

1. WHEN ピアへのダイヤル試行が回復可能な輸送層エラーで失敗した場合 THEN セッションは `max_retries` が示す追加回数まで再試行を行うこと。
2. IF `retry_backoff_ms` が設定されている場合 THEN 連続する各再試行の間で指定ミリ秒だけ待機すること。
3. WHEN 全再試行が失敗し元のエラーが解消しない場合 AND 設定された回数を使い切った場合 THEN 総試行回数と最終エラー要約を含む失敗結果を返すこと。

### Requirement 2

**User Story:** CLI オペレーターとして、再試行の進捗を把握できる構造化フィードバックがほしい。そうすれば成功へ向けた状況をリアルタイムで判断できる。

#### Acceptance Criteria

1. WHEN 新たな再試行が開始される場合 THEN 試行番号・経過時間・対象ピア ID を含む構造化ログを出力すること。
2. IF CLI のイベントストリームが有効化されている場合 THEN 各再試行につき `PeerEvent::DialRetry`（または同等）を発行し、ピア識別子・現在の試行回数・次の待機時間を伝えること。
3. WHEN ダイヤルが最終的に成功する場合 THEN 実際に行われた再試行回数と成功試行で計測した RTT を結果に含めること。

### Requirement 3

**User Story:** プラットフォームのメンテナとして、CLI ドキュメントにリトライ仕様が明文化されていてほしい。そうすれば利用者がフラグ設定を理解し適切に調整できる。

#### Acceptance Criteria

1. WHEN CLI ドキュメントをビルドする場合 THEN リトライ上限、待機時間、ログ出力の挙動を説明する更新を含めること。
2. IF 既定値に変更がない場合 THEN デフォルト値とコマンドでの上書き方法を明記すること。
3. WHEN 将来リトライ関連フラグを変更する場合 THEN ドキュメントも同時に更新する運用ルールを記述すること。

## Non-Functional Requirements

### Code Architecture and Modularity
- **単一責務**: リトライ制御ロジックをダイヤルワークフロー内に閉じ込めつつ、既存のログ／イベント経路を介して外部へ通知する。
- **モジュール性**: 待機・再試行ユーティリティを共通化し、将来の外向き接続でも再利用可能とする。
- **依存管理**: 新たな非同期ランタイム依存を追加せず、既存の Tokio プリミティブを活用する。
- **明確なインターフェース**: 既存のセッション設定 API を後方互換で拡張し、既存呼び出し側に破壊的変更を与えない。

### Performance
- リトライ導入によるオーバーヘッドは、正常系において `max_retries * retry_backoff_ms + 25ms` を超えないこと。

### Security
- リトライログにおいては機密性の高いピアトークンを出力せず、匿名化された識別情報のみを記録すること。

### Reliability
- 永続的エラーや認証失敗は再試行対象から除外し、不必要なループを避けること。

### Usability
- CLI 出力は再試行状況を要約表示し、デフォルト設定でも冗長にならないよう配慮すること。

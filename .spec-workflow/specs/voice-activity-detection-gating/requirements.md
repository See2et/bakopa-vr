# Requirements Document

## Introduction

BakopaVR の Rust サイドカーに音声アクティビティ検出（VAD）ゲーティングを導入し、会話者以外のバックグラウンドノイズを自動的に抑えつつ、低遅延な音声配信を維持する。しきい値制御やヒステリシス、参加者ごとの状態共有を備え、Unity クライアントが空間オーディオのミキシングや視覚的フィードバックに活用できるようにする。

## Alignment with Product Vision

本機能は、"マイクロソーシャルスペース" でクリアな会話を実現し、手動モデレーションの負担を減らすことでプロダクトの目標を強化する。ゲーティング処理をサイドカー側で完結させることで「ピア優先の信頼性」という製品原則を支援し、VAD 状態遷移を Bloom や診断に公開することで可観測性の向上にも寄与する。ノイズの少ないストリームは、小規模セッションを最小限の運用負担で実行できるかを検証する MVP の成功指標に直結する。

## Requirements

### Requirement 1

**User Story:** BakopaVR の参加者として、発話していないときに自動的に自分のマイクをミュートしてもらい、他の参加者がアイドルノイズに邪魔されないようにしたい。

#### Acceptance Criteria

1. WHEN ローカル音声入力が設定された VAD しきい値をサイレンスタイムアウトの間下回り続けた場合 THEN サイドカーは 50ms 以内に送出オーディオフレームを最大で −60dB まで減衰させる。
2. IF VAD 分類器がローカルストリームを `inactive` と判定した場合 THEN 分類器が `active` へ戻るまで、サイドカーは iroh ストリーム上でゼロ以外の PCM ペイロードを送信しない。
3. WHEN 分類器が `inactive` から `active` へ遷移し、かつクライアントが強制ミュート状態でない場合 THEN サイドカーは 30ms 以内に通常レベルの音声送出を再開し、Unity クライアントへ `active` 状態を通知する。

### Requirement 2

**User Story:** リモート参加者として、他の参加者の話者状態が正確に同期されることで、空間オーディオのミキシングやアバターの口パクが実際の発話に追従するようにしたい。

#### Acceptance Criteria

1. WHEN サイドカーが任意の参加者の VAD 状態を更新した場合 THEN Unity クライアントが 100ms 以内に取得できるよう、`participant_id`、`vad_state`、タイムスタンプを含むセッションメッセージを送信する。
2. IF Unity クライアントが任意の参加者について `silent` ステータスを受信した場合 THEN 新しい `active` ステータスを受信するかホストがチャネルを強制開放するまで、その参加者のミックスゲインを少なくとも 20dB 下げ、口パクアニメーションを停止する。

### Requirement 3

**User Story:** イベントホストとして、音楽再生やアクセシビリティなどの例外に対応できるよう VAD の挙動を制御し、透過的に状況を把握したい。

#### Acceptance Criteria

1. WHEN ホストが Bloom もしくは Unity のモデレーターパネルで任意参加者の「常時オープン」を切り替えた場合 THEN サイドカーは 1 フレーム以内に該当参加者の VAD 減衰をバイパスする。
2. IF VAD 推論が失敗した場合（例: 分類器エラー、CPU 使用率が設定上限を超過） THEN サイドカーはパススルー音声にフォールバックし、エラーコードとストリーム識別子を含む診断イベントを発行する。
3. WHEN 参加者がホスト操作により `force-muted` 状態へ移行した場合 THEN サイドカーはホストが解除するまでゼロ化フレームを送信し続け、分類器の出力に関わらず `force-muted` 状態をブロードキャストする。

## Non-Functional Requirements

### Code Architecture and Modularity
- **Single Responsibility Principle**: VAD 解析は専用の Rust モジュールで実装し、セッションパイプラインへは最小限のインターフェースを提供する。
- **Modular Design**: しきい値設定、分類器実行、状態ブロードキャストを別コンポーネントとして分離し、各コンポーネントにユニットテストを整備する。
- **Dependency Management**: WebRTC VAD（または同等ライブラリ）はサイドカー側でラップ／ベンダリングし、Unity から実装詳細を隠蔽する。
- **Clear Interfaces**: Unity への VAD 更新は既存の gRPC IPC スキーマ内で表現し、分類器内部実装と直接結合しない。

### Performance
- VAD 処理によるエンドツーエンド遅延の増加は音声フレームあたり 10ms 以下、CPU 使用率は Windows リファレンス VR クライアントで 5% 未満とする。
- サイレント状態の参加者について、ネットワーク帯域使用量を常時送信と比較して少なくとも 30% 削減する。

### Security
- VAD 状態メッセージには生の音声サンプルを含めず、サイドカーと Bloom／ピア間の既存暗号化を維持する。
- 診断ログにはタイムスタンプとストリーム ID のみを記録し、個人を特定できる情報は含めない。

### Reliability
- サイドカーは分類器初期化に失敗した場合、パススルーへ移行する前に最大 3 回リトライを行う。
- 各アクティブ参加者について、VAD 健全性を 5 秒間隔で示すウォッチドッグメトリクスを出力する。

### Usability
- 参加者は VAD 状態の変化から 200ms 以内にマイク状態インジケータ（例: アイコン点灯）を視認できる。
- ホストはモデレーター用ツールから参加者ごとに VAD バイパスを切り替えられる。
